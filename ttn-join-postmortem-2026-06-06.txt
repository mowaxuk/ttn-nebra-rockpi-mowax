ROOT CAUSE ANALYSIS
===================

There were three interlocking failure modes. Any one alone would have
been survivable. Together they stretched a simple re-join into a 4-hour
problem.

1. NVS NOT PERSISTING DEVNONCE ON JOIN FAILURE (primary bug)
-------------------------------------------------------------
The old firmware called persist.saveSession() only on successful join.
When the device power-cycled, NVS was at the DevNonce from the LAST
SUCCESSFUL join (approximately 43, from the June 5 evening session).

Every subsequent cold boot loaded this value and started trying from ~44.
But TTN's floor was already above 44 (from accumulated prior sessions and
then worsened by the burst at 10:47-10:48). The device would exhaust its
backoff cycle, apparently "give up" (or get reset), and reload the same
NVS value, starting the same cycle again.

This is why the CLAUDE.md session notes described jreqs "wrapping" at
DevNonces 43→61→43 — that wasn't a watchdog reset, it was NVS not
advancing. Each boot restarted from the same floor.

THE FIX: Call persist.saveSession(&node) after every activateOTAA() call,
success or failure. NVS now advances on each attempt. Each cold boot
starts above the previous cold boot's highest attempt.

2. RAPID RETRIES EXHAUSTING 869.525 MHz DUTY CYCLE
---------------------------------------------------
Without exponential backoff, the device sent jreqs as fast as
activateOTAA() returned — effectively back-to-back (7 in ~45 seconds
at 10:47-10:48 UTC). The EU868 RX2 window is 869.525 MHz at 1% duty
cycle. A JoinAccept on SF12/BW125 is ~2.5s of airtime. The budget is
approximately 36s per hour.

The burst used 7 × ~2.5s = 17.5s of airtime in under 1 minute. The
gateway's duty cycle tracker saw all subsequent JoinAccept scheduling
requests as over-budget and emitted ns.down.join.schedule.fail for every
one. The join server accepted the jreq and generated a JoinAccept; the
gateway just couldn't transmit it.

This created the cruel situation visible in the TTN device live data:
js.join.accept appearing, as.up.join.forward appearing, but the device
sitting confused with no response.

THE FIX: Exponential backoff: 30s → 60s → 120s → 240s → 300s cap.
At 300s between attempts, one JoinAccept every 5 minutes — well within
the EU868 1% duty cycle budget (~1 per 250s theoretical minimum).

3. BURST AT 10:47-10:48 ADVANCED FLOOR WITHOUT DEVICE RECEIVING JOINSACCEPT
-----------------------------------------------------------------------------
This is the compounding effect. When a jreq succeeds (js.join.accept),
TTN's join server increments its accepted-DevNonce counter for that
DevEUI. This happens even if the device never receives the JoinAccept
(ns.down.join.schedule.fail).

The 7-jreq burst advanced TTN's floor by 7. But NVS stayed at the pre-
burst value (old firmware, no save). So after the fix was applied, the
device started incrementing from the pre-burst NVS value — which was now
7 MORE steps below the new higher floor. The burst didn't help; it made
things slightly worse by raising the target.

After the fix, the device needed to increment DevNonce from its NVS value
all the way up through the new elevated floor, one attempt per 300s.
That accounts for the 2h46m from fix deployment (~11:00 UTC) to join
(~13:47 UTC): approximately 33 attempts × 300s = ~165 minutes, which
lands the device at the right DevNonce to cross TTN's floor.


WHY TTN'S FLOOR WAS SO HIGH TO BEGIN WITH
==========================================
The TTN join server maintains a per-DevEUI DevNonce floor permanently —
it survives device deletion, re-registration, and application changes.
Prior debugging sessions on this device (before 6 June) had run the
device through many jreq cycles. Each accepted join request advances the
floor. Even rejected jreqs (dev_nonce_too_small) do NOT advance the floor.

The June 5 evening join used DevNonce ~43, which was accepted. This set
the floor to ~43. But 6-7 jreqs accepted in the 10:47-10:48 burst raised
it by 7 again. Then subsequent accepted jreqs (if any from earlier June 6
attempts that fell in the gap in available logs) may have raised it further.

By the time the fix was working correctly (NVS incrementing, backoff
spacing out attempts), the floor was high enough to require ~33 × 300s
of attempts to cross. The device crossed it at ~13:47 UTC.


WHAT FIXED IT (CONFIRMED)
==========================
Two changes to heltec-otaa/src/main.cpp:

Change 1 — NVS save on failure:
  Before:
    if (state == RADIOLIB_LORAWAN_NEW_SESSION || ...)
        persist.saveSession(&node);
    // failure path: no save → NVS DevNonce unchanged

  After:
    if (join success) {
        persist.saveSession(&node);
    } else {
        persist.saveSession(&node);  // NVS DevNonce advances even on fail
        nextAttempt = millis() + backoff;
    }

Change 2 — Exponential backoff:
  uint32_t shift = (attempt - 1) < 4 ? (attempt - 1) : 3;
  uint32_t backoff = 30000UL * (1UL << shift);  // 30s, 60s, 120s, 240s
  if (backoff > 300000UL) backoff = 300000UL;   // cap at 300s

LESSONS
=======
1. Always call persist.saveSession() after EVERY activateOTAA() call.
   The return code doesn't matter; the DevNonce must advance.

2. EU868 Join requires exponential backoff, minimum 300s at cap.
   1% duty cycle on 869.525 MHz is unforgiving. A 45-second burst of
   jreqs wastes 4 hours of join attempts.

3. TTN's DevNonce floor is permanent and accumulates across all sessions.
   Every accepted jreq raises it. Every rejected jreq (dev_nonce_too_small)
   does NOT raise it. This asymmetry is crucial: sending jreqs that get
   accepted-but-device-misses is worse than sending none at all.

4. ns.down.join.schedule.fail is not a join failure at the join server.
   js.join.accept + ns.down.join.schedule.fail = floor advanced, device
   gets nothing. This is the worst possible outcome per join attempt.

5. RTC RAM vs NVS: RST button preserves RTC RAM (session restored instantly,
   no RF, no new jreq). Power cycle clears RTC RAM (cold boot, NVS only,
   full RF join required). Never confuse the two — they have completely
   different DevNonce behaviour.

6. DevNonce floor cannot be reset by deleting/re-registering the device.
   The only reset is TTN support intervention or waiting for the counter
   to wrap (65535 → 0 is a 16-bit counter; wrapping would require tens
   of thousands of jreqs — not realistic).
